# CLAUDE.md

Conventions and constraints for working on this repo. Read this before any task.

## Project

**Name:** FlowOps (working title)
**Purpose:** Asset management platform for water distribution, wastewater collection, and storm drainage. Cityworks alternative — focused, modern, open standards.
**Status:** Greenfield. No existing code to preserve.
**Owner:** Kyle Cahill (kathail). Solo dev, sellable product target.

## Stack — non-negotiable

- **Backend:** Python 3.12, Flask 3.x, SQLAlchemy 2.x (typed), Alembic, Pydantic v2 for request/response schemas
- **Database:** PostgreSQL 16 + PostGIS 3.4. No SQLite, ever. Spatial queries are first-class.
- **API:** Flask blueprints, JSON only, REST. OpenAPI spec generated via `apispec`.
- **Auth:** Flask-Login + Argon2 hashing. Session cookies, `Secure`, `HttpOnly`, `SameSite=Lax`. CSRF via Flask-WTF on state-changing requests. Plan for OIDC later but do not build it yet.
- **Frontend:** React 18 + Vite + TypeScript + Tailwind. MapLibre GL JS for mapping. TanStack Query for server state. Zustand for client state.
- **Tiles:** `pg_tileserv` (Crunchy Data) running as a sidecar. Vector tiles only.
- **Files:** S3-compatible storage via `boto3`. Local dev uses MinIO in Docker. Production uses Backblaze B2 or Cloudflare R2 (configurable via env).
- **Background jobs:** RQ (Redis Queue). Simpler than Celery, sufficient for MVP.
- **Tests:** pytest, pytest-flask, factory-boy, faker. Frontend: Vitest + React Testing Library + Playwright for e2e.
- **Lint/format:** `ruff` (lint + format) for Python. ESLint + Prettier for TS. `pre-commit` runs both.

## Style rules — read carefully

### Python

- Use type hints everywhere. `from __future__ import annotations` at the top of every module.
- SQLAlchemy: declarative `Mapped[...]` style, not legacy `Column(...)` syntax.
- Pydantic models live in `app/schemas/`. SQLAlchemy models in `app/models/`. Never mix them.
- Config via `pydantic-settings` reading from env. No hardcoded values.
- Logging via stdlib `logging` configured once in `app/logging.py`. Structured JSON logs in production.
- Errors: raise typed exceptions from `app/errors.py`. A central error handler converts to JSON responses.
- No `print()`. No bare `except:`. No `# type: ignore` without a comment explaining why.

### TypeScript

- `strict: true` in tsconfig. No `any` without an `// eslint-disable-next-line` and a reason.
- Components: function components only, named exports, one component per file.
- API client generated from OpenAPI spec via `openapi-typescript`. Do not hand-write request types.
- Tailwind only. No CSS modules, no styled-components, no inline styles except dynamic values.

### Database

- Every table has `id BIGSERIAL PRIMARY KEY`, `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()`. Exception: append-only event tables (e.g., `audit_log`) use the event timestamp (e.g., `occurred_at`) and omit `updated_at`.
- All FKs explicit and named. All indexes named. Never rely on autogenerated names.
- Spatial columns use `GEOMETRY(GEOMETRY, 4326)` and have a GIST index.
- JSONB columns have GIN indexes if queried.
- Migrations are reviewed before applying. Never edit a migration after it's been applied to any environment.
- Soft delete via `deleted_at TIMESTAMPTZ NULL`. Hard delete only by admin action.

### Git

- Branch per feature: `feat/asset-crud`, `fix/wo-status-bug`. No work on `main` directly.
- Conventional commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`.
- Squash merge to `main`. PR description must reference the SPEC.md section being implemented.
- Never commit `.env`, `*.pem`, anything in `secrets/`. `.gitignore` is comprehensive — extend it, don't bypass it.

## Environments

- **Local dev:** `docker compose up` brings the full stack. Postgres on 5432, Redis on 6379, MinIO on 9000, pg_tileserv on 7800. Backend on 5000, frontend on 5173.
- **Railway:** Production target. Hobby plan. Postgres + Redis as Railway services. Backend and frontend as separate services. Files on B2.
- **Migrations:** Run via `flask db upgrade` on deploy. Railway start command runs migrations before serving.

## Hard rules — do not violate

1. **Never write code that bypasses PostGIS for spatial work.** No haversine in Python, no bounding-box-in-application logic. Use `ST_*` functions.
2. **Never store passwords in plaintext or with MD5/SHA-1/bcrypt.** Argon2 only.
3. **Never expose internal IDs in URLs for tenant data.** Use `asset_uid` (the human-readable code) in routes, look up `id` server-side.
4. **Never trust client-supplied `tenant_id`.** Tenancy is derived from the authenticated user's session.
5. **Never run a destructive query without a backup or transaction.** Migrations that drop columns require a two-phase rollout (deploy code that ignores it, then drop).
6. **Never add a dependency without justifying it in the PR.** Every package is a liability.
7. **Never write a feature without a test.** Backend: unit + integration. Frontend: at minimum a component render test, e2e for critical flows.
8. **Never commit a TODO without a tracking issue number.** `# TODO(#42): ...`

## Working style

- Read `docs/SPEC.md` before starting any feature. The spec is the source of truth.
- If the spec is ambiguous, raise the question in the PR description. Do not guess.
- Small PRs. One concept per PR. If a PR description needs more than three bullet points to explain, split it.
- Tests in the same PR as the code they cover. Never "tests in a follow-up."
- When in doubt about an architectural call, default to the simpler option and add a comment explaining the choice.

## Multi-tenancy

This is a sellable product. Multi-tenancy is built in from day one.

- Every domain table has `tenant_id BIGINT NOT NULL REFERENCES tenant(id)`.
- Every query filters by `tenant_id`. This is enforced via a SQLAlchemy session-level event listener (`app/extensions.py::tenant_filter`).
- Cross-tenant data leaks are P0 bugs.
- Row-level security in Postgres is set up but the application layer is the primary enforcement.
